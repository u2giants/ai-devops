# Designflow Tracking API Reference

This service handles production tracking, licensing tracking, lead-time
calculation, and sample tracking.

In deployed environments callers should normally use the BFF:

```text
https://api.designflow.app/api/tracking/<tracking-path>
```

Direct local base:

```text
http://localhost:5002
```

## Auth

Most routes require a Designflow JWT. The backend accepts the token in
`Authorization`. When called through the BFF, the BFF sends the user JWT as
`X-User-Authorization`, and the service remaps it to `Authorization`.

Common roles:

```text
admin, sourcing_manager, sales, designer, vendor, production
```

# Architecture — designflow-tracking

## Overview

A single Express service exposing the Designflow tracking domain. It is private (Cloud Run
`--no-allow-unauthenticated`) and is reached only through the BFF.

```
Browser (Angular: designflow-frontend)
  → designflow-bff  (terminates user JWT, attaches Cloud Run OIDC token,
                     forwards user JWT as x-user-authorization)
  → designflow-tracking  (/api/tracking/* → service root paths)
  → PostgreSQL (Sequelize)        + DigitalOcean Spaces (sample photos)
```

`index.js` first copies `x-user-authorization` back onto `authorization` so route guards
(`helpers/auth.js authRole`) see the user JWT even though the BFF replaced `authorization` with
the Cloud Run OIDC token.

## Request layering

```
routes/<x>.router.js     authRole([...]) gate, path → controller
controllers/<x>.controller.js   thin: unwrap req, call model, map errors to HTTP status
models/<x>.model.js      domain logic, callback style: (req, result) => result(err, data)
models/db.js → models/db/*.js   Sequelize models (schema source of truth)
```

`routes/index.js` mounts: `/` (lib), `/lic` (licensing), `/lead-time`, `/sample`, `/api` (health).

## Persistence and migrations

- `models/db.js` builds the Sequelize instance, populates the exported `db` object
  **synchronously** (model definitions don't need a live connection), then awaits
  `authenticate()` → `sync()` → `applySampleMigrations()`.
- `sequelize.sync()` creates brand-new tables from the models. Additive column changes go in
  the idempotent `applySampleMigrations()` block (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS`,
  snake_case only). There is no separate migration CLI.

## Sample tracking

The sample module is modelled on production tracking (its own tables) rather than licensing
(shared `itemHeader`), because `itemHeader` is owned by the item-master sync.

**Tables** (`models/db/`): `sample`, `sample_event`, `sample_comments`, `sample_attachment`,
`sample_factory_group`, `sample_box`, `sample_shipment_item`. Associations are wired in
`models/db/init-models.js`.

**Lifecycle / state machine** (`helpers/sampleStatus.js`): a sample has an `origin`
(factory_request / factory_offer / usa_bought) that sets its `direction`:

- inbound: `requested → shipped → ningbo_checked_in → ningbo_checked_out → nyc_checked_in`
- outbound: `nyc_requested → nyc_checked_out → ningbo_checked_in → ningbo_checked_out → sent_to_factory`
- shared extras (any time): `cancelled`, `on_hold`, `damaged`, `approved`

`recordEvent` derives the next status from `(direction, current status, office, event_type)`,
writes it under an optimistic guard (`WHERE status=<from_status>`, 0 rows → 409), and appends a
`sample_event` row. Boxes carry their own status independent of member samples; adding a
direction-mismatched sample to a box is skipped and reported (warn, not block).

**Grouping**: `sample.factory_group_id_fk` (factory shipment group) and `sample.box_id_fk`
(current box cache) plus `sample_shipment_item` for per-leg box membership (a box can bundle
samples from multiple factory groups).

**Integration links**: `sample.item_id_fk` (→ `itemHeader`) and `sample.prod_order_no_fk`
(→ `ProdOrderHeader`) let licensing/production attach samples to their records.

**Photos**: presigned-URL upload to DigitalOcean Spaces (`helpers/digitalocean.js`):
`POST /sample/getPreSignedUrl` → browser PUT → `POST /sample/setObjectPublic` →
`POST /sample/photos` stores the public URL in `sample_attachment`.

## Constraints

- Service must stay private; the BFF is the only ingress.
- FKs are application-enforced (not DB constraints); the model layer owns cascade/integrity
  (e.g. `Sample.remove` deletes child events/comments/photos/shipment-items).
- `npm test` runs before boot in every `start:*` script.

# Configuration — designflow-tracking

Environment is loaded by `config/env.js` from `.env.<NODE_ENV>` (e.g. `.env.localhost`). In
deployed environments the values come from Cloud Build / Secret Manager (see
[`deployment.md`](deployment.md)), not from a committed `.env`.

## Variables

| Variable | Purpose | Source (prod) | Required dev | Required prod |
|---|---|---|---|---|
| `DB_HOST` | Postgres host | Secret Manager | yes | yes |
| `DB_PORT` | Postgres port | Secret Manager | yes | yes |
| `DB_USER` | Postgres user | Secret Manager | yes | yes |
| `DB_PASSWORD` | Postgres password | Secret Manager | yes | yes |
| `DB_NAME` | Postgres database | Secret Manager | yes | yes |
| `SCHEMA` | Postgres schema (e.g. `designflow`) | `--set-env-vars` | yes | yes |
| `JWT_PUBLIC_KEY` | RS256 public key for verifying user JWTs | Secret Manager | yes | yes |
| `JWT_PRIVATE_KEY` | RS256 private key | Secret Manager | if signing locally | yes |
| `WEB_URL` | Frontend URL (startup log / context) | `--set-env-vars` | optional | yes |
| `API_ADDRESS` | API address (startup log / context) | `--set-env-vars` | optional | yes |
| `PORT` | Listen port (default `5002`) | Cloud Run `--port` | optional | yes |
| `DO_ACCESS_KEY` | DigitalOcean Spaces key | Secret Manager (`--set-secrets`) | for photo upload | yes |
| `DO_SECRET_KEY` | DigitalOcean Spaces secret | Secret Manager (`--set-secrets`) | for photo upload | yes |
| `DO_ENDPOINT_URL` | Spaces endpoint | `--set-env-vars` from `_DO_ENDPOINT_URL` | for photo upload | yes |
| `DO_BUCKET_NAME` | Spaces bucket | `--set-env-vars` from `_DO_BUCKET_NAME` | for photo upload | yes |
| `DO_PUBLIC_URL` | Spaces public CDN base | `--set-env-vars` from `_DO_PUBLIC_URL` | for photo upload | yes |

`config/db.config.js` throws at startup if any of `DB_HOST/DB_PORT/DB_USER/DB_PASSWORD/DB_NAME`
is missing. The `DO_*` vars are read by `config/digitalocean.config.js` and only needed by the
sample photo-upload endpoints.

> **Note (DigitalOcean):** `cloudbuild.yaml` provisions the DO keys via Secret Manager
> (`DO_ACCESS_KEY`, `DO_SECRET_KEY` — the `deployer@` runtime SA already has project-level
> `secretmanager.secretAccessor`) and the bucket/endpoint via `--set-env-vars`, with
> `_DO_ENDPOINT_URL`/`_DO_BUCKET_NAME`/`_DO_PUBLIC_URL` **defaulted in `cloudbuild.yaml`** to the
> shared `dflowbucket` Spaces values (same in sandbox + prod). No trigger change is required.
> Locally, put the `DO_*` values in `.env.localhost`.

# Deployment — designflow-tracking

Derived from `Dockerfile` and `cloudbuild.yaml`.

## Pipeline

Cloud Build (`cloudbuild.yaml`) runs three steps:

1. **Build** — `docker build -t ${_IMAGE} --build-arg NODE_ENV=${_APP_ENV} .`
   (`Dockerfile`: `node:lts-alpine`, installs `imagemagick`, `yarn` install, `CMD yarn
   start:$NODE_ENV` — which runs `npm test` then starts the server).
2. **Push** — pushes `${_IMAGE}` to Artifact Registry. Image path:
   `${_REGION}-docker.pkg.dev/${PROJECT_ID}/${_ARTIFACT_REPO_NAME}/${_APP_ENV}/${_APP_NAME}:${BUILD_ID}-${COMMIT_SHA}`.
3. **Deploy** — `gcloud run deploy ${_SERVICE_NAME}` with `--no-allow-unauthenticated`
   (private), `--vpc-connector`, `--service-account`, `--memory`, `--region`, `--port`,
   `--concurrency`, `--max-instances`.

## Secrets and env at deploy

- `--set-secrets`: `DB_USER`, `DB_PASSWORD`, `DB_NAME`, `DB_HOST`, `DB_PORT`,
  `JWT_PRIVATE_KEY`, `JWT_PUBLIC_KEY`, `DO_ACCESS_KEY`, `DO_SECRET_KEY` (all from Secret Manager
  `projects/$PROJECT_NUMBER/...`; the `deployer@` runtime SA has project-level secretAccessor).
- `--set-env-vars`: `WEB_URL`, `API_ADDRESS`, `SCHEMA`, `DO_ENDPOINT_URL`, `DO_BUCKET_NAME`,
  `DO_PUBLIC_URL` (the `DO_*` ones come from `_DO_*` trigger substitutions).
- `_DO_ENDPOINT_URL`/`_DO_BUCKET_NAME`/`_DO_PUBLIC_URL` are **defaulted in `cloudbuild.yaml`**
  to the shared `dflowbucket` Spaces values (identical in sandbox + prod, verified against the
  core services), so no trigger change is needed. A trigger-level `_DO_*` substitution still
  overrides the defaults if a different bucket is ever required.

## Trigger and substitutions

All `_*` substitutions (`_REGION`, `_SERVICE_NAME`, `_APP_ENV`, `_APP_NAME`,
`_SERVICE_ACCOUNT_NAME`, `_PORT`, `_CONCURRENCY`, `_MAX_INSTANCES`, `_VPC_CONNECTOR`,
`_PLATFORM`, `_MEMORY`, `_ARTIFACT_REPO_NAME`, `_WEB_URL`, `_API_ADDRESS`, `_DB_SCHEMA`) are
provided by the **Cloud Build trigger** (one per environment), not hard-coded in the repo. The
build runs when the branch the trigger watches receives a push.

## Rollback

Redeploy a previous image tag to the Cloud Run service:

```
gcloud run deploy <service> --image <region>-docker.pkg.dev/<project>/<repo>/<env>/<app>:<previous-build-id>-<sha> --region <region>
```

## Access / SSH

The service is private; the **BFF** is the only ingress (it attaches the Cloud Run OIDC token).
There is no SSH-based deploy or admin path — deployment is image-based via Cloud Build/Cloud Run.

## Pre-deploy gate

`yarn start:$NODE_ENV` runs `npm test` first; a failing test suite stops the container from
starting. Keep the suite green before pushing the deploy branch.
# Development — designflow-tracking

## Prerequisites

- Node.js (CommonJS; the Docker image uses `node:lts-alpine`)
- A reachable PostgreSQL instance
- `imagemagick` (installed in the Docker image; only needed if you exercise image features locally)

## Setup

```bash
npm install
```

Create `.env.localhost` in the repo root (loaded by `config/env.js` via `.env.<NODE_ENV>`):

```
DB_HOST=...
DB_PORT=5432
DB_USER=...
DB_PASSWORD=...
DB_NAME=...
SCHEMA=designflow
JWT_PUBLIC_KEY=...        # RS256 public key (PEM)
JWT_PRIVATE_KEY=...       # if needed locally
# Sample photo upload (DigitalOcean Spaces) — optional locally:
DO_ACCESS_KEY=...
DO_SECRET_KEY=...
DO_ENDPOINT_URL=...
DO_BUCKET_NAME=...
DO_PUBLIC_URL=...
PORT=5002
```

`config/db.config.js` throws on startup if any `DB_*` var is missing. See
[`configuration.md`](configuration.md) for the full list.

## Run

```bash
npm run start                 # NODE_ENV=localhost (runs npm test first, then nodemon)
npm run start:sandbox         # NODE_ENV=sandbox
npm run start:sandbox-albert  # NODE_ENV=sandbox-albert
```

The server logs the port (default 5002), env, schema, and the `.env` file in use. On boot it
also caches `LicenseFeedBacks`, `LicensingTime`, and builds lead-time fuzzy-match maps.

## Test

```bash
npm test                      # all Jest unit tests
npx jest tests/unit/sample.model.test.js   # one file
npx jest --watch              # watch mode
```

Tests mock Sequelize and helpers — they do not need a database:

```js
jest.mock('../../models/db', () => ({ sample: { findByPk: jest.fn(), update: jest.fn(), /* ... */ } }));
jest.mock('../../helpers/auth', () => ({ getUserInfo: jest.fn(async () => ({ user_id: 1, name: 'X' })) }));
```

Always add or extend a unit test when you add/change a function. Source-string guard tests live
in `tests/unit/db.migration.test.js` (table/column presence, init ordering, snake_case DDL).

## Debugging tips

- 403/401 from a route → the user JWT role isn't in the route's `authRole([...])` list, or the
  `x-user-authorization` header wasn't forwarded by the BFF.
- 409 from `POST /sample/events` → optimistic concurrency: the sample's status changed since you
  read it (or an illegal transition). Re-read and retry.
- Startup crash reading `db.LicenseFeedBacks` → ensure `models/db.js` still populates `db`
  (Object.assign) **before** the first `await` (see AGENTS.md §9).

# Designflow PLM System Architecture

This document maps the Designflow product lifecycle management system across the
local repositories under `D:\repos\dflow`. It is written from the code as it
exists in this workspace, not from the GitHub repo names alone.

The important correction to the informal "five repo" model is that the runtime
PLM system has six active application/service repos locally:

- `designflow-frontend`
- `designflow-bff`
- `designflow-backend`
- `designflow-item-master`
- `designflow-tracking`
- `designflow-data-syncing`

The original five named repos are the main subject, but `designflow-item-master`
is an active service in the request path. The frontend calls it through the BFF,
and the BFF has a first-class `item_master` service mapping. Any complete system
map should include it.

## Executive Summary

Designflow is a Cloud Run based PLM application. The browser runs a static
Angular SPA. The SPA does not call private backend services directly in deployed
environments; it calls the BFF at `api.*.designflow.app`. The BFF proxies each
request to one of the private domain services, adding a Cloud Run OIDC token for
service-to-service invocation while preserving the user's Designflow JWT in
`X-User-Authorization`.

Most services use:

- Node.js CommonJS
- Express
- PostgreSQL
- Sequelize
- `SCHEMA` to select the logical database schema
- RS256 Designflow JWTs for user identity and role checks

The main persistent coupling between services is the shared PostgreSQL schema.
The HTTP/API coupling is intentionally centralized through the BFF.

## Repository Responsibilities

### `designflow-frontend`

The frontend is the Angular 21 SPA. It owns:

- UI routing and feature modules

- RFQ calculation behavior in the browser
- User JWT storage in `localStorage`
- HTTP interceptors for auth, loading, and error reporting
- Microsoft SSO browser flow before exchanging the Azure ID token for a
  Designflow JWT

The deployed frontend calls the BFF, not individual backend services directly.
The environment files define these service bases:

- `environment.url` -> `/api/core`
- `environment.item_master` -> `/api/item_master`
- `environment.tracking` -> `/api/tracking`
- `environment.data_sync` -> `/api/data_sync`

Major user-facing modules:

- RFQ
- Item Library
- Item Detail
- Art Piece management
- Standardized Products
- Production Tracking
- Licensing Tracking
- Sample Tracking
- Lead Time Calculator
- Admin / Editor
- User Management and impersonation
- Notifications
- HTS / tariff / AI classification UI

### `designflow-bff`

The BFF is the public API gateway for the SPA. It owns:

- `/health`
- `/api/:service/*`
- service key mapping
- Cloud Run OIDC identity token generation
- original user JWT forwarding
- deployed CORS configuration

Service keys:

- `core` -> `designflow-backend`
- `item_master` -> `designflow-item-master`
- `tracking` -> `designflow-tracking`
- `data_sync` -> `designflow-data-syncing`

The BFF strips the service prefix before proxying. For example:

```text
Browser:
  GET /api/tracking/sample/list

BFF target:
  GET /sample/list on designflow-tracking
```

Cloud Run OIDC audience must be the backend service `.run.app` URL, not the
custom `api.*.designflow.app` domain. This is why the BFF environment variables
must use backend `.run.app` URLs.

## Request and Auth Flow

```mermaid
sequenceDiagram
  participant U as User / Browser
  participant FE as Angular SPA
  participant BFF as designflow-bff
  participant SVC as Private backend service
  participant DB as PostgreSQL

  U->>FE: Uses PLM feature
  FE->>BFF: HTTP request with Authorization: Bearer DesignflowJWT
  BFF->>BFF: Resolve service key from /api/:service/*
  BFF->>BFF: Fetch Google OIDC token for backend .run.app URL
  BFF->>SVC: Authorization: Bearer CloudRunOIDC
  BFF->>SVC: X-User-Authorization: Bearer DesignflowJWT
  SVC->>SVC: Copy X-User-Authorization back to Authorization
  SVC->>SVC: authRole validates RS256 Designflow JWT and role
  SVC->>DB: Sequelize query/update
  DB-->>SVC: Rows
  SVC-->>BFF: JSON response
  BFF-->>FE: JSON response
```

This dual-token pattern is the key to understanding deployed authentication:

- Cloud Run sees the OIDC token and allows the BFF to invoke the private service.
- Express route guards see the restored user JWT and enforce product roles.

the same database schema. Shared model families include:

- Users, roles, permissions
- Vendors/factories
- Customers
- Merch groups
- UDF tables/elements/components

## End-to-End Flows

### Sign-In and Identity

Password login and Microsoft SSO both end by storing a Designflow RS256 JWT in
the browser. For SSO, the browser obtains an Azure ID token, posts it to
`/api/core/auth/azure`, and the core backend verifies Azure JWKS before signing a
Designflow JWT.

Every subsequent browser request sends the Designflow JWT to the BFF.

### RFQ and Vendor Quoting

RFQ is primarily a frontend + core-backend workflow:

1. Frontend loads grid layout and RFQ rows from core backend.
2. Frontend applies AG Grid column definitions and calculation value setters.
3. User edits RFQ item fields or vendor quote fields.
4. Frontend recalculates landed cost, margins, duty, container cost, and sales
   pricing where relevant.
5. Core backend persists RFQ item/vendor changes.
6. Brevo may send quote/signup/request emails.
7. Vendor users access filtered RFQ data through the same BFF/core path.

RFQ also reaches into other domains:

- Factory country data is used by tariff calculations.
- Item images/attachments may be stored in DigitalOcean Spaces.
- Grid layouts and view states are shared system metadata.
- Standardized products can export/import to RFQ.

### Item Master and Art Piece

Item Library, Item Detail, and Art Piece screens are frontend modules that route
through `/api/item_master`.

The item-master service owns the active API boundary for these screens. It reads
and writes item master tables, while data-sync can populate or refresh item data
from Coldlion into the same schema family.

### Production and Licensing Tracking

Production and licensing tracking route through `/api/tracking`.

Production tracking works with production order tables and factory-facing order
date updates. Licensing tracking works with licensing status/milestone/grouping
data and item license images. The tracking service builds startup caches for
licensing/factory lead-time calculations.

### Sample Tracking

Sample Tracking is a newer tracking module under `/api/tracking/sample`. It has
its own tables and state machine rather than using `itemHeader` directly as the
primary record. It links to the broader PLM through:

- `item_id_fk`
- `prod_order_no_fk`
- factory/group references
- sample events
- sample boxes
- sample attachments/photos

The server is authoritative for sample status transitions.

### Lead-Time Calculation

The frontend calls `/api/tracking/lead-time/*`. The tracking service uses design
team time, licensing time, factory time, merch group descriptions, and fuzzy
matching helpers to calculate lead time and needed dates.

### HTS, Tariffs, and AI Classification

The frontend duty-rate tools call core backend admin/openrouter/cross routes.

Core backend:

- proxies OpenRouter chat/model calls
- proxies CBP CROSS search
- looks up bundled MFN HTS rates
- manages Section 301 upload/lookup state
- manages Trump 2025 country-rate settings
- stores AI/tariff app settings

The model suggests classification. The system derives actual rates from
authoritative lookup sources and admin-maintained settings.

### Coldlion Sync

Data Sync is the Coldlion integration layer. It is API-key protected and intended
for service/system usage.

It pulls external data into the same database schema used by the application
services. That makes sync a background/source-of-truth feeder for item,
merch-group, vendor/customer, production-order, and prepack-association data.

## Operational Notes

- The app has no single OpenAPI contract across all services.
- Many routes are internal Angular-shaped APIs and return domain rows or
  grid-specific payloads.
- BFF routes are stable as a gateway, but service contracts remain mixed
  legacy/new style.
- The shared database is the strongest cross-service coupling.
- Additive schema changes are commonly handled by startup Sequelize sync and
  idempotent inline migrations rather than a dedicated migration repository.
- JWT is stored in browser `localStorage`, not an HTTP-only cookie.
- DigitalOcean direct browser upload uses a presigned PUT followed by an
  explicit `setObjectPublic` call.

| Lead-time | `/lead-time/*` | Design team, licensing, and factory time lookups plus calculation. |
| Sample tracking | `/sample/*` | Samples, events, comments, boxes, groups, photos, and status state machine. |

## Inter-Service Couplings

- Production and licensing rows depend on item, vendor, merch group, and
  production order data that may also be used by core backend and item-master.
- Data-sync imports production order and external reference data from Coldlion
  that tracking screens consume.
- Sample tracking intentionally has its own tables, but links back to the wider
  PLM with `item_id_fk` and `prod_order_no_fk`.
- Sample photos use DigitalOcean Spaces through a presigned URL flow similar to
  core backend media workflows.
- The frontend reaches this service only through BFF `/api/tracking`, and the BFF
  forwards the Designflow JWT through `X-User-Authorization`.
